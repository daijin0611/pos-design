## 工作量分布

### 后端

| 类型 | 占比 | 说明 |
|------|------|------|
| 可复用现有接口 | ~20% | VIP/Order/KPI 数据和接口已存在，仅需包装移动端查询 |
| 新增表 + 标准 CRUD | ~30% | 预约、积分、评分等新模块 |
| 复杂业务逻辑 | ~35% | 预约系统、回访提醒、积分规则、手机收银（占大头） |
| 第三方对接 | ~15% | 微信 OAuth / 短信 / OSS / 模板消息 / 银行 |

### 前端

| 类型 | 占比 | 说明 |
|------|------|------|
| H5 项目基础设施 | ~16% | 从零搭建：骨架、路由、布局、授权、组件库适配 |
| 列表/详情页 | ~40% | 大量移动端页面（消费记录、预约、工资、排名等） |
| 复杂交互页面 | ~25% | 预约流程、手机收银、积分兑换、图片对比 |
| PC 端增量 | ~14% | 图片录入、订货、对账 |
| 地图/扫码/图表 | ~5% | 腾讯地图、ECharts 移动适配、JS-SDK 调用 |

## 风险与关键依赖

| 风险项 | 影响 | 建议 |
|--------|------|------|
| 公众号服务号资质审核 | 阻塞全部 H5 工作 | 第一天就提交认证，审核需 1-7 天 |
| 现有工资计算逻辑完整性 | E1/A2 可能不够 | 提前与业务方确认提成规则 |
| 订货宝 API 文档 | P2 工作量不确定 | 尽早获取文档评估 |
| 银行对接 API | P3 各银行差异大 | 确认合作银行后评估 |
| JS-SDK 兼容性 | 扫码/定位部分机型有问题 | 准备降级方案 |

---

## 结论

**前端 > 后端**（1.4 : 1），核心原因是 H5 项目从零搭建，而后端有大量可复用。

- 后端重量级工作集中在**预约、积分、手机收银**三个需求（占后端 40%）
- 前端重量级工作集中在**项目基础设施 + 预约流程 + 手机收银**（占前端 35%）
- 管理端 5 项需求后端工作量最小（合计 7d），因现有数据表和查询逻辑可大量复用

建议 **2 后端 + 2 前端**并行，第一、二阶段完成后可释放 1 名后端。
