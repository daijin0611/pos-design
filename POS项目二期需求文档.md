# POS 项目二期需求文档（细化版）

> 基于原始 23 项需求，结合一期已有系统功能，深入挖掘隐含的子需求、前置依赖和跨模块关联需求。
>
> **技术方案**：微信公众号 H5 网页（非小程序），一套 Vue3 + Vant 移动端应用，按角色分发路由。

---

## 技术架构

```
微信公众号
├── 关注公众号 → 粉丝沉淀（可群发推文、模板消息触达）
├── 菜单栏入口 → H5 页面
├── 扫码/分享链接 → H5 页面
│
└── H5 移动端（Vue3 + Vant，独立于现有 PC 端）
    ├── /member/*    会员端路由
    ├── /staff/*     员工端路由
    └── /admin/*     管理端路由
    │
    └── 后端 API（复用现有 Spring Boot 服务，新增移动端接口）
```

**选型理由**：
- 三角色共用一套代码，按登录角色分发路由，维护成本最低
- 团队已有 Vue3 经验（现有 PC 端 `lld_cms` 即 Vue3 + Element Plus），移动端仅需切换 Vant 组件库
- 公众号关注即获客，业务功能（查余额/预约/优惠券）自然驱动用户关注，形成「线下消费 → 关注公众号 → H5 互动 → 内容推送 → 复购」闭环
- 无需审核发布，更新即时生效，不受小程序包体限制

| 编号 | 子需求 | 说明 |
|------|--------|------|
| G1.1 | 公众号注册与配置 | 注册服务号（需企业资质），配置服务器域名、JS-SDK 安全域名、网页授权域名 |

| 编号 | 子需求 | 说明 |
|------|--------|------|
| G2.1 | 微信网页授权 | 用户在微信内打开 H5 → 自动触发 OAuth2.0 静默授权或显式授权 → 获取 OpenID |

| G2.4 | 管理端登录 | 管理人员登录需角色校验（仅管理员角色可访问管理端路由） |
| G2.5 | Token 多端互通 | PC 端和 H5 端共享用户身份上下文，JWT 支持多端签发与验证 |

| 编号 | 子需求 | 说明 |

### M3. 收银验证码（一级）

**原始需求**：收银时需会员输入密码或验证码确认，验证码通过服务号推送

**业务背景**：会员在门店消费时（PC 收银或员工 H5 收银），使用会员卡余额支付需经会员本人确认。通过公众号服务号推送验证码，会员口头告知收银员或自行在 H5 页面输入，确保消费安全，防止未经授权的余额扣款。

| 编号 | 子需求 | 说明 |
|------|--------|------|
| M3.1 | 验证码推送触发 | 收银端发起余额扣款时，系统自动生成 6 位数字验证码，通过公众号模板消息推送到会员微信 |
| M3.2 | 验证码校验 | 收银端输入会员口述的验证码，后端校验正确后放行扣款；验证码 5 分钟内有效，验证成功后自动失效 |
| M3.3 | 密码支付（可选） | 会员也可选择设置支付密码，收银时直接输入密码确认（替代验证码方式） |
| M3.4 | 推送消息模板 | 验证码模板消息包含：消费门店、消费金额、验证码、有效期提示 |
| M3.5 | 安全策略 | 同一会员 60 秒内限发 1 次，每日上限 10 次；连续 3 次验证失败锁定 30 分钟 |
| M3.6 | 免密额度（可选） | 可配置小额免密额度（如 ≤ 100 元无需验证码），在管理端统一配置 |

**关联需求**：E7（店长手机收银 — 触发验证码）、PC 收银端（触发验证码）、G3.2（公众号模板消息推送）、G2.2（会员账号体系）

**业务背景**：会员接受修复/调理服务后，服务效果和过程需记录存档。传统方式由 PC 端录入，效率低且不及时。通过员工 H5 端即时录入，实现服务完毕即记录，提升健康档案的完整性和时效性。

| E6.8 | 健康档案数据表 | 新建 `health_record` 表：记录ID、会员ID、员工ID、门店ID、项目ID、服务日期、回访截止日期、调理次数、详情描述、状态 |
| E6.9 | 图片关联 | 新建 `health_record_image` 表：记录ID、图片URL、图片分类（修复前/后、调理前/后）、上传时间（关联 G4 文件存储） |

**关联需求**：E2（回访提醒 — 录入后自动生成待回访）、P1（PC 端图片管理 — 数据互通）、M10（会员端查看健康档案）、G4（文件存储服务）、现有 `vip_info`、`order_detail` 表
